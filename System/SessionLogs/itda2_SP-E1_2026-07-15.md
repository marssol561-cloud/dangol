## 발생 에러 + 처리 결과
1. **마이그레이션 이력 불일치**: `supabase migration list` 확인 결과 원격 `supabase_migrations.schema_migrations`에 001~009가 타임스탬프 포맷(`20260621085710` 등)으로만 기록되어 있고, 로컬 파일명(001~009)과 매칭되지 않아 `db push`가 001부터 재적용을 시도(테이블 이미 존재 → 실패 위험). → `supabase migration repair --status reverted <9개 타임스탬프>` + `--status applied 001..009`로 이력 북키핑만 정합화(스키마/데이터 변경 없음, 대시보드 SQL 아님 — CLI 표준 기능). `--dry-run`으로 010만 적용 대상임을 확인 후 `db push` 실행. **미해결 아님 — 해결 완료.**
2. **테스트 헬퍼 customers insert 실패**: 신규 테스트 3개 파일의 `createCustomer` 헬퍼가 마이그레이션 009에서 추가된 `unsub_token`(NOT NULL) 컬럼을 누락 → insert가 null 반환. → 3개 파일 모두 `unsub_token` 필드 추가로 수정, 전체 green. **해결 완료.**
3. **[범위 밖, 미수정] 기존 테스트 22개 파일 사전 실패 발견**: `npx vitest run` 전체 실행 시 `returningCoupon.test.ts`, `returningMatch.test.ts`, `rls_customer.test.ts`, `rls_sp4.test.ts`, `segment.test.ts`, `sendGuard.test.ts` 등 다수가 동일 원인(customers insert에 `unsub_token` 누락)으로 이미 실패 중이었음. `git diff main...feat/spe1-event-infra`로 확인 결과 본 스프린트에서 건드리지 않은 파일들이며, 마이그레이션 009(본 스프린트 이전 병합)의 NOT NULL 제약이 원인 — SP-E1 착수 이전부터 존재하던 회귀. `dangolDb.test.ts`의 `schema_version` 기대값도 '007'로 낡아 있음(실제는 '010'). 지시서 파일 범위(§FILE STRUCTURE) 밖이라 **수정하지 않음** — 지시자 판단 필요.
