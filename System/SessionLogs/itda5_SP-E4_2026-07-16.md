## 설계 판단사항 (지시서 여백을 코드로 직접 해결 — CONSTRAINT 조항에 따름)
1. **`issueEventCoupon`의 `db` 인자화**: 기존 `lib/coupons.ts`의 `insertCoupon`류는 내부에서 `getServerClient()`를 직접 호출해 테스트 시 DB 클라이언트 주입이 불가능한 구조. 지시서가 명시적으로 `issueEventCoupon(db, {...})` 시그니처를 지정했고, `lib/events.ts`(SP-E1~E3)의 모든 함수가 이미 `db`를 첫 인자로 받는 관례를 따르므로 이를 그대로 적용 — 기존 `insertCoupon` 자체는 변경하지 않고 별도 함수로 신설
2. **`approveParticipation`의 동시성 가드**: 지시서는 "이미 approved면 멱등 반환"만 명시했으나, 두 스태프가 동시에 같은 참여를 승인 클릭하는 경쟁 상황에서 쿠폰이 2번 발급될 위험이 있어 `UPDATE ... WHERE status='pending'`으로 원자적 가드 후 영향받은 행이 0이면(경쟁에서 짐) 함수를 1회 재귀 재실행해 승자의 결과를 그대로 반환하도록 보강 — DONE CRITERIA "idempotent" 요건을 동시성 하에서도 실제로 보장하기 위한 필수 보정
3. **cross-store 접근 → `not_found`로 매핑**: 지시서 "require it belongs to staffStoreLinkId"의 반환 형태가 명시되지 않아, SP-E2의 `getEventDetail`/`updateEvent`가 cross-store를 `null`/`notFound`로 처리하는 기존 규약을 그대로 따름(라우트에서 404) — `not_pending`(409)과 명확히 구분
4. **`listPendingApprovals`의 "오늘" 판정 = KST 자정 기준**: 지시서가 "same-day only"만 명시하고 타임존을 지정하지 않아, 국내 서비스(리붐단골)의 실제 영업일 기준인 KST(UTC+9)로 자정을 계산하는 순수 함수(`startOfTodayKST`, 비export)를 구현 — 서버 배포 리전(UTC)에 의존하지 않도록 함
5. **customer_tags 분리 구분자**: 지시서 DETAILED SPEC이 "comma/·"를 명시 → `/[,·]/`로 split 후 trim+빈 문자열 제거. `condition`이 비어있으면(null/공백) 태그 생성 스킵(지시서 5번 항목 "Skip if condition empty" 그대로)
6. **승인 대기 화면의 "toast"**: 본 레포에 공용 Toast 컴포넌트가 없어(grep 결과 0건) 새로 만들면 파일 범위(app/coupon-use/page.tsx 단일 MOD) 밖의 공유 컴포넌트 신설이 되므로, 기존 `result` 배너와 동일하게 로컬 상태(`toast`) + 3초 자동 소멸 인라인 배너로 구현 — 공유 컴포넌트 미신설
