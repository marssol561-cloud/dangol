## 설계 판단사항 (지시서 여백을 코드로 직접 해결 — CONSTRAINT 조항에 따름)
1. **`event-status`의 `browser_token` 조회 방식**: 지시서는 "GET with query browser_token"이라 명시했으나, `dangol_bt` 쿠키는 `app/api/customers/route.ts` 8단계에서 `httpOnly: true`로 설정되어 있어 클라이언트 JS(`document.cookie`)로 값을 읽을 수 없음(페이지의 기존 B3 친구초대 링크 코드가 이 방식으로 값을 읽으려다 실패하는 기존 버그를 확인했으나 본 스프린트 파일 범위 밖이라 미수정). 따라서 명시적 `browser_token`/`channel`+`identifier` 쿼리 파라미터가 없을 때는 `/api/checkin`과 동일하게 서버가 `cookies()`로 `dangol_bt`를 직접 읽어 조회 — 쿼리 파라미터 지원 자체는 유지(테스트·유연성 목적), 실제 클라이언트 호출은 쿠키 자동 첨부에 의존
2. **`getParticipationStatus`가 "활성 이벤트의 참여"가 아닌 "매장 내 최근 참여"를 조회**: 지시서 문구는 "their participation in the store's ACTIVE event"이나, `issue_cap` 도달로 승인 처리 자체가 이벤트를 `resolveStoreEvent` 기준 `closed`로 전환시키는 경우(기둥 로직: `approvedCount>=issue_cap`) 승인 직후 정확히 그 순간 폴링이 참여자를 찾지 못하는 결함이 발생함. `event_participations`를 `store_link_id`+`customer_id`로 최신 1건 조회하도록 구현해 이 결함을 방지 — DONE CRITERIA의 "Poll event-status; when status becomes 'approved', show the issued coupon"을 실제로 만족시키기 위한 필수 보정
3. **`app/r/[code]/page.tsx` 구조 — 오버레이 게이트 패턴**: 평시 `step` 상태 머신(`'loading'|'b1'|'b2'|'b3'|'done'|'error'`)과 그 전이 로직은 단 한 줄도 건드리지 않음. 대신 `activeEvent`+`eventGateDismissed`+`eventJoinStep` 3개 상태로 별도 게이트를 앞단에 배치, `showEventGate`가 참일 때만 이벤트 B1/B2/B3를 렌더링하고 거짓이면 기존 평시 렌더링 경로로 그대로 진입. 이 설계로 "평시 flow unchanged" 요건을 코드 diff 수준에서 검증 가능하게 함(기존 useEffect 삽입분 4줄 제외)
4. **`event-join` 요청에 `visit_purpose` 미포함**: DETAILED SPEC의 EVENT B2 화면 명세가 identifier+name+동의만 언급하고 방문목적 선택지를 포함하지 않아, 이벤트 참여 폼에서는 방문목적을 묻지 않음(`createParticipation`은 `visitPurpose` optional 파라미터를 지원하므로 향후 필요 시 인자만 추가하면 됨)
